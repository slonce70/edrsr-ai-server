# План оптимизации egress без миграции

Цель: существенно снизить исходящий трафик из БД (Supabase Postgres) и API, не меняя архитектуру и поведение для пользователя. Ниже — проверенный по коду план внедрения, с приоритетами, деталями реализации и критериями проверки.

Важно: часть нижеприведённых идей всплывала в стороннем анализе, но здесь указано, что действительно относится к нашему проекту (по фактическому коду).

## 1) Где «течёт» трафик (верифицировано по коду)

- Крупнейшая причина: многократные «толстые» выборки задания с полными текстами дел и финальным отчётом:
  - `server/services/dbService.js::getJob()` — всегда подтягивает:
    - все ссылки из `job_links` с полем `content` (полный текст дела);
    - `job_results.analysis_text` (финальный отчёт).
  - `server/services/dbService.js::updateJobStatus()` — после `UPDATE` вызывает `getJob(jobId)` и возвращает «толстый» объект на каждом шаге прогресса.
  - `server/routes/index.js::GET /status/:id` — возвращает «толстый» объект по умолчанию (используется и результатами, и попапом/ручной проверкой).
  - `server/routes/index.js::GET /jobs/last` — отдаёт «толстый» объект (для попапа это избыточно).

- WebSocket: при завершении задачи (`status === 'completed'`) отправляется «толстый» объект (через `updatedJob`, полученный на предыдущем шаге из `getJob()`). На промежуточных шагах — отправляется «лёгкий», но толстая выборка всё равно уже произошла.

- Список обработанных URL: `GET /processed-urls` отдаёт все уникальные URL пользователя (растущий список), хотя клиенту чаще нужно проверить уникальность текущей страницы.

- Что уже «лёгкое» и ок:
  - `GET /jobs` — возвращает только «лёгкие» поля (id/status/progress/created_at/title), без ссылок и контента.
  - Чат — лимитирован `LIMIT 50`.

## 2) Принципы оптимизации

- Никогда не выбирать большие поля (link `content`, `analysis_text`), если они не используются немедленно.
- По умолчанию — «лёгкие» ответы; «тяжёлые» — только по явному запросу.
- Снизить частоту и размеры «дорогих» чтений во время прогресса; оставить «толстые» чтения только в точках, где действительно нужно (например, перед итоговой выдачей на странице результатов).
- Для «уникальных ссылок» — проверять членство (membership) для набора URL текущей страницы, а не выгружать весь список.

## 3) План внедрения (итерациями)

### Итерация A — убрать лишние «толстые» чтения на прогрессе (самый большой эффект)

1. `dbService.updateJobStatus`:
   - Заменить «`UPDATE ...; return getJob(jobId)`» на «`UPDATE ... RETURNING`» только нужных полей:
     - `id, status, progress, processed_links, total_links, created_at, updated_at, title, prompt`.
   - Возвращать «лёгкий» объект; никаких ссылок и `analysis_text`.

2. `server/routes/index.js::updateStatus(...)` (используется воркером):
   - Использовать «лёгкий» объект из п.1, формировать WS‑payload только из нужных полей.
   - Для `status === 'completed'` — по-прежнему отправлять WS‑событие, но без «толстых» полей. Если клиенту требуется полный объект — пусть запрашивает его отдельно (см. Итерация B).

Критерии готовности:
- Промежуточные апдейты больше не инициируют чтение `job_links.content` и `job_results.analysis_text`.
- WS‑сообщения в процессе работы содержат только «лёгкие» поля.

Статус: ВЫПОЛНЕНО
- dbService.updateJobStatus теперь использует `UPDATE ... RETURNING` и возвращает «лёгкие» поля.
- В updateStatus «лёгкий» payload отправляется на каждом апдейте; «тяжёлый» объект выбирается ровно один раз на `completed` (для совместимости с текущим расширением).

### Итерация B — сделать «лёгкие» ответы по умолчанию

3. `GET /status/:id`:
   - По умолчанию — возвращать «лёгкий» объект задания (без ссылок‑контента и без `analysis_text`).
   - Добавить явные флаги/варианты получения «тяжёлых» полей:
     - `?include=analysis` — вернуть также `analysis_text`;
     - `?include=links` — вернуть ссылки без `content` (только `url, status, decision_date`);
     - отдельный эндпоинт для выгрузки контента ссылок (см. п.5), чтобы не тянуть их всегда.

4. `GET /jobs/last`:
   - Возвращать только «лёгкий» объект (использовать то же, что и `GET /jobs`/`getRecentJobs`). Попапу не требуется контент/анализ.

Критерии готовности:
- Попап/ручная проверка статуса получают быстрые «лёгкие» ответы.
- Страница результатов продолжит работать: анализ будет получен явно (см. п.6), а контент ссылок — только при необходимости (см. п.5).

Статус: ВЫПОЛНЕНО
- `GET /status/:id` по умолчанию возвращает «лёгкий» объект.
- Добавлены флаги: `?include=analysis`, `?include=links` (без контента ссылок).
- Расширение уже ориентируется на лёгкий ответ + отдельные эндпоинты для тяжёлых данных.

### Итерация C — выборка «тяжёлых» данных только по требованию

5. Новый эндпоинт для контента ссылок задания:
   - `GET /jobs/:id/links-content` — выдаёт список `{ url, content }` только для ссылок со статусом `processed`. Подходит для кнопки «Скачать TXT».
   - Альтернативно — сделать этот эндпоинт стриминговым или отдавать как файл.

6. Новый эндпоинт для анализа:
   - `GET /jobs/:id/analysis` — возвращает только `analysis_text`.

Изменения на стороне расширения (минимальные):
- `results.html/js`:
  - При загрузке страницы получать `analysis` через `GET /jobs/:id/analysis`.
  - Кнопка «Скачать TXT» — вызывать `/jobs/:id/links-content` и формировать файл; больше не полагаться на `job.links[].content` в основном ответе `/status`.

Критерии готовности:
- Страница результатов загружается быстро без «толстого» `/status`.
- Контент ссылок уходит по сети только по клику «Скачать TXT».

Статус: ВЫПОЛНЕНО ЧАСТИЧНО
- Реализованы эндпоинты: `GET /jobs/:id/analysis`, `GET /jobs/:id/links-content`.
- Результат‑страница теперь:
  - при открытии запрашивает `status?light=true` и `analysis` отдельно;
  - для кнопки «Скачать TXT» запрашивает `links-content` on‑demand.
- WebSocket‑обновления на `completed` теперь обогащаются анализом на стороне service worker, если он отсутствует в событии.

### Итерация D — оптимизация «уникальных» ссылок

7. Новый эндпоинт для проверки членства (membership):
   - `POST /urls/processed-check` с телом `{ urls: string[] }` → ответ `{ processed: string[] }`.
   - Реализуется через один `SELECT url FROM job_links WHERE status='processed' AND url = ANY($1)`.
   - Индекс: добавить `CREATE INDEX IF NOT EXISTS idx_job_links_url ON job_links(url)` (ускорит membership).

8. Обновить использование в расширении (popup, content):
   - Вместо `GET /processed-urls` (всё) — вызывать `POST /urls/processed-check` для текущей страницы (обычно десятки/сотни URL, а не тысячи из истории).
   - Service Worker: добавлен хендлер `API_CHECK_PROCESSED` для фонової перевірки membership з Bearer‑токеном.

Критерии готовности:
- Убрана передача длинного общего списка обработанных URL.
- Проверка «уникальных» работает быстрее и дешевле.

Статус: ВЫПОЛНЕНО
- Реализован эндпоинт `POST /urls/processed-check` + індекс `idx_job_links_url`.
- `extension/bg.js`: добавлен обработчик `API_CHECK_PROCESSED` → возвращает `{ success, urls: processed }`.
- `extension/popup.js` и `extension/content.js`: переведены на membership‑проверку по списку URL страницы.

### Итерация E — финишная полировка

9. WS‑события, не требующие тяжёлых данных (например, переименование/завершение задания):
   - Отправлять «лёгкий» payload без ссылок и анализа; при необходимости клиенты подтягивают тяжёлые данные по отдельным эндпоинтам.

Статус: ВЫПОЛНЕНО
- Обновление названия задания — «лёгкий» payload.
- Завершение задачи — «лёгкий» payload; service worker при необходимости дозагружает `analysis`.

10. Пагинация и лимиты:
   - Запретить «бесконечные» ответы: `GET /jobs` теперь ограничен `JOBS_MAX_LIMIT` (по умолчанию 100), даже если `limit=all`.

Статус: ВЫПОЛНЕНО
- Сервер ограничивает выдачу `/jobs` согласно `JOBS_MAX_LIMIT`.

11. HTTP‑сжатие API (gzip/br):
   - Включить `compression` middleware в Express для снижения egress API (не влияет на egress БД, но полезно для внешнего трафика к расширению и админке).

Статус: ВЫПОЛНЕНО ЧАСТИЧНО
- Подключён `compression` в `server/server.js` (по умолчанию >1KB).
- Добавлена зависимость `compression` в `server/package.json` (требуется `npm install` на деплое).

Критерии готовности:
- Все массовые события и списки ограничены и/или «лёгкие» по умолчанию.
- API‑ответы компактны на проводе.

## 4) Что из «другого анализа» подтверждено/не подтверждено

Подтверждено и включено в план:
- Убирать «толстые» выборки на каждом апдейте прогресса (через `RETURNING`).
- Делать «лёгкий» `/status` по умолчанию; «тяжёлые» данные — отдельными запросами.
- Не слать «толстые» объекты по WS без необходимости.
- Оптимизировать уникальность ссылок через membership‑проверку вместо выгрузки полного списка.

Уточнения по нашему коду:
- `GET /jobs` уже «лёгкий» (возвращает базовые поля, без ссылок и контента).
- История чата уже ограничена `LIMIT 50` — существенного вклада в egress не даёт.
- Сжатие контента «при сохранении в БД» не уменьшит egress БД, если всё равно приходится читать и передавать контент — наш упор на «не выбирать» контент до запроса.

## 5) Порядок работ и оценка эффекта

Рекомендуемый порядок (по убыванию эффекта/сложности):
1) Итерация A — убрать «толстые» чтения на прогрессе (самый большой выигрыш, низкий риск).
2) Итерация B — «лёгкий» `/status`, «тяжёлые» поля только по флагам.
3) Итерация C — отдельные эндпоинты для анализа и контента ссылок + адаптация `results.js`.
4) Итерация D — membership‑проверка `processed-urls`.
5) Итерация E — пагинация/лимиты, WS‑полировка, HTTP‑compression.

Ожидаемая суммарная экономия egress:
- −50–90% только за счёт A + B (исключение повторных «толстых» выборок).
- Дополнительно −10–30% за счёт C + D + E.

## 6) Тест‑план и валидация

- Метрики до/после (минимум на уровне приложения):
  - Логгировать размер сериализованного ответа (грубая оценка): `Buffer.byteLength(JSON.stringify(payload))` для `/status`, `/jobs/last`, WS «completed».
  - Считать количество вызовов `getJob()` в проде и заменить на «лёгкие» аналоги.
- Инструменты БД (если доступно в Supabase): `pg_stat_statements` для топ‑запросов по объёму/времени.
- Интеграционные проверки:
  - `npm run dev`, `curl /api/health/{light,full}`.
  - Обычный цикл: старт задания на 50–300 ссылок → наблюдать прогресс (WS) → открыть результаты → скачать TXT.
  - `npm run test:memory` — убедиться, что память стабильна после дробления ответов.

## 7) Риски и обратная совместимость

- Изменения API: результаты/попап потребуют небольших правок для явной загрузки анализа и контента ссылок. Это локальные изменения в расширении (`extension/results.js`, `extension/bg.js`).
- Кэширование на клиенте: сейчас кэшируется «лёгкое» состояние — это поможет уменьшить повторные запросы; при отделении анализа имеет смысл кэшировать и его.
- Безопасность: новые эндпоинты должны сохранять текущую модель авторизации (Bearer с Supabase Auth) и user‑scoping.

---

Этот план нацелен на максимальный выигрыш в egress за счёт отказа от нецелевых «толстых» SELECT и разделения данных на «лёгкие»/«тяжёлые». Он минимально инвазивен, укладывается в существующую архитектуру и не требует миграции с Supabase.
