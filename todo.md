# EDRSR‑AI: Аудит и план исправлений

Документ фиксирует полный анализ репозитория (сервер + расширение) и поэтапный план работ с подзадачами. На этом этапе изменений в код не вносится — только TODO.

## 1) Краткое резюме
- Монорепозиторий: Node.js (Express + PostgreSQL) и Chrome Extension (MV3), Supabase для аутентификации, Gemini для ИИ‑аналитики, очередь + worker_threads, WebSocket‑уведомления.
- Архитектура зрелая: отдельные слои (маршруты, middleware, сервисы, БД, воркеры), индексы БД, миграции через connection.js, кэш «parsed_cases», RLS‑SQL.
- Найдены критичные риски безопасности (URL‑валидация, доступ к внутренним/системным эндпоинтам, публичный health‑check к Gemini, XSS в расширении) и ряд логических/UX‑багов.

## 2) Инвентаризация репозитория
- Корень:
  - `package.json` — скрипты для сервера/линта/формата/сборки расширения.
  - `scripts/` — build extension, apply RLS.
- Сервер (`server/`):
  - `index.js` (старт), `server.js` (Express + WS), `routes/` (API), `middleware/`, `database/connection.js` (Pool, миграции, индексы), `services/dbService.js`, `scraper.js`, `worker.js`, `gemini.js`, `config.js`, `utils.js`.
  - `sql/` — RLS скрипты.
- Расширение (`extension/`):
  - `manifest.json`, `bg.js`, `popup.html/js`, `content.js`, `results.html/js`, `config.js`, `auth.js`, сторонние минифицированные библиотеки.

## 3) Ключевые находки (риски и уязвимости)
1. SSRF/валидация URL (критично)
   - В местах фильтрации URL используется `includes('reyestr.court.gov.ua')`. Возможна подмена хоста/поддомены/схема. Риск SSRF и утечки Cookie заголовка в `got()`.
   - Нужно строго валидировать `new URL(url)` → `hostname === 'reyestr.court.gov.ua'` и путь `/Review/...`.

2. Внутренний эндпоинт можно обойти (критично)
   - `/api/internal/process-queue` допускает `x-internal-request`. Подделать просто.
   - Требуется защита: только localhost И/ИЛИ секретный токен/роль администратора.

3. Публичный health‑check бьёт в Gemini (критично)
   - `/api/health/full` без авторизации вызывает `generateContent('ping')` → расход квоты/денег и DDoS‑поверхность.
   - Нужна авторизация/кэширование/мок, либо полностью закрыть для не‑админов, плюс rate limit.

4. Системные маршруты доступны обычным пользователям (критично)
   - Эндпоинты типа `/api/system/stats`, `/api/queue/clear`, `/api/workers/*` не ограничены админом.
   - Требуется `requireAdmin` и/или отдельные права + аудит.

5. XSS‑поверхность в расширении (критично)
   - В `results.js` и чате Markdown рендерится через `marked.parse(...)` с последующей вставкой `innerHTML`. Нет явной санации (DOMPurify). Ссылки уже ограничены, но этого недостаточно. Возможны скриптовые инъекции через контент отчёта/истории.
   - Нужна санация HTML, запрет изображений/скриптов, строгий список тегов/атрибутов.

6. Раскрытие деталей ошибок (важно)
   - Глобальный `errorHandler` возвращает клиенту `details: err.message`. В продакшне это лишнее и опасно.

7. Логический баг в `/api/collect` (важно)
   - После валидации используется исходный `links`, а не `validLinks`. В очередь попадает «грязный» массив — рассинхрон и потенциальные ошибки в воркере.

8. Недостаточная валидация DTO и лимитов (важно)
   - Нужно ограничить количество ссылок в запросе, длины полей, структуру объектов (серверная валидация).

9. CORS/заголовки (важно)
   - Сейчас общий `cors()`. Лучше добавить `helmet`, отключить `x-powered-by`, ограничить origins/методы/заголовки.

10. Дублирующие функции и шум логов
   - Дублируется `testGeminiConnection` (в `gemini.js` и `utils.js`). В продакшне лог‑уровень и стек‑трейсы стоит зафлагать по `NODE_ENV`/`LOG_LEVEL`.

## 4) Прочие наблюдения
- Хорошая дисциплина БД: индексы, миграции, блокировки/лейзинг для очередей, кеш дел `parsed_cases` c upsert.
- WS‑аутентификация реализована корректно (auth + subscribe). Рекоммендовано переаутентифицировать WS при рефреше токена — частично сделано в расширении.
- CSP для `/admin` задан, но с `unsafe-inline` (приемлемо для статических страниц, но лучше минимизировать инлайн‑скрипты).
- Стоимость: `maxOutputTokens: 65536` может быть дорогой режим по умолчанию — стоит пересмотреть.

## 7) Критерии приёмки (выборка)
- Запрос с невалидным URL/лишними полями в `/collect` получает 400/422, ничего не попадает в очередь.
- Ссылка на сторонний домен/поддомен отбрасывается на сервере (и в `downloadAll`).
- `/api/health/full` доступен только админу; обращение «с улицы» возвращает 401/403; вызов не жжёт квоту ИИ.
- Обычный пользователь не может получить доступ к `/api/system/*`, `/api/queue/clear`, `/api/workers/*`.
- В расширении любые попытки XSS в отчёте/чате не рендерят опасный HTML/скрипты.
- В продакшне ошибки без внутренних деталей, а логи на сервере содержат всё нужное для отладки.
